# Revit API Patterns

This skill is the "stuff that isn't obvious from reading the .NET BCL" reference for this plugin. If you're adding a button, sync handler, or Revit document mutation, start here.

## The threading rule (the #1 thing to know)

**Revit API calls must run on Revit's API thread, inside an active API context.** A WPF `Button.Click` handler is *not* on that thread. Calling Revit API directly from WPF will either throw `InvalidOperationException: Attempt to modify document outside of transaction` or — worse — silently misbehave.

The bridge is `IExternalEventHandler` + `ExternalEvent`:

```csharp
// In the UserControl constructor, register the handler once:
_exportHandler = new ExportHandler();
_exportEvent = ExternalEvent.Create(_exportHandler);

// In a WPF click handler, set fields then raise:
private void ExportButton_Click(object sender, RoutedEventArgs e)
{
    _exportHandler.SelectedProjectId = _selectedProject?.Id;
    _exportHandler.Direction = SyncDirection.Update;
    _exportEvent.Raise();   // Revit invokes Execute() when API context is next available
}
```

Inside `Execute(UIApplication app)` you're safe to touch `app.ActiveUIDocument.Document`, run a `FilteredElementCollector`, open a `Transaction`, etc. See `HVAKR.Revit.UI/ExportHandler.cs` for the canonical example.

`Execute` is **synchronous** — if you have async work (HTTP calls to HVAKR), you're stuck doing sync-over-async (`.GetAwaiter().GetResult()`) inside the handler. That's fine here because you're already on Revit's thread and not blocking the UI; don't let the pattern leak elsewhere.

## `IExternalCommand` vs `IExternalApplication` vs `IExternalEventHandler`

| Interface                | Lifecycle                                         | Use for                                                                 |
| ------------------------ | ------------------------------------------------- | ----------------------------------------------------------------------- |
| `IExternalApplication`   | Called once on Revit startup/shutdown             | Ribbon setup, dockable pane registration (`HVAKR.Revit/App.cs`)            |
| `IExternalCommand`       | Called each ribbon-button click                   | Short, synchronous ops — e.g. the pane toggle (`ToggleDockablePaneCommand.cs`) |
| `IExternalEventHandler`  | Called when you `.Raise()` an `ExternalEvent`     | Bridging WPF → Revit API (`ExportHandler.cs`)                              |

Plugin entry is declared in `HVAKR.addin` (generated by Inno Setup) as `<FullClassName>HVAKR.Revit.App</FullClassName>` with `Type="Application"`.

## Transactions

Every document mutation must be inside a transaction:

```csharp
using (var t = new Transaction(document, "Add HVAKR Spaces"))
{
    t.Start();
    // ... create/modify elements ...
    t.Commit();
}
```

- The name you pass (`"Add HVAKR Spaces"`) shows up in Revit's Undo stack. Be descriptive.
- Nest with `SubTransaction` only if you have a good reason. We don't have one yet.
- If an exception is thrown mid-transaction, Revit will roll back on dispose. Don't catch-and-swallow inside the transaction.

## Dockable pane

The pane GUID (`App.DockablePaneGuid`, `3c649293-cdeb-4415-bbf1-d037ed56ba4e`) is a **stable identifier** — Revit stores per-user dock position under it. Changing the GUID orphans every user's existing layout. Don't change it.

Registration happens once in `App.OnStartup`:

```csharp
application.RegisterDockablePane(
    new DockablePaneId(App.DockablePaneGuid),
    "HVAKR",
    new DockablePaneProvider(providerData)
);
```

`Command.Execute` toggles visibility:

```csharp
var pane = commandData.Application.GetDockablePane(new DockablePaneId(App.DockablePaneGuid));
if (pane.IsShown()) pane.Hide(); else pane.Show();
```

## Units

Revit stores **everything length in decimal feet** and **angles in radians**, regardless of the project's display unit. Use `UnitUtils` at the boundary, never inline arithmetic:

```csharp
// Reading from Revit, sending to HVAKR (which wants inches + degrees)
double inches    = UnitUtils.ConvertFromInternalUnits(feet,   UnitTypeId.Inches);
double degrees   = UnitUtils.ConvertFromInternalUnits(radians, UnitTypeId.Degrees);

// Writing to Revit, values came from HVAKR
double feet      = UnitUtils.ConvertToInternalUnits(inches, UnitTypeId.Inches);
double radians   = UnitUtils.ConvertToInternalUnits(degrees, UnitTypeId.Degrees);
```

`UnitTypeId` lives in `Autodesk.Revit.DB` — it's the Revit 2022+ replacement for the old `DisplayUnitType` enum. Don't import `DisplayUnitType`.

## Filtered element collection

```csharp
var spaces = new FilteredElementCollector(document)
    .OfCategory(BuiltInCategory.OST_MEPSpaces)
    .WhereElementIsNotElementType()
    .Cast<Space>()
    .ToList();
```

- `WhereElementIsNotElementType()` is almost always what you want — without it you get type definitions too.
- `.Cast<T>()` throws on mismatch; use `.OfType<T>()` if a category might contain mixed element types.
- Keep the pipeline lazy when possible, but `.ToList()` early if you're going to iterate twice.

Common categories in this plugin:

| Category                         | Element type        |
| -------------------------------- | ------------------- |
| `OST_MEPSpaces`                  | `Space`             |
| `OST_Levels`                     | `Level`             |
| `OST_SpaceSeparationLines`       | `CurveElement`      |

## Geometry primitives

- `XYZ(double x, double y, double z)` — feet. Use `XYZ.Zero`, `XYZ.BasisX/Y/Z` when you can.
- `UV(double u, double v)` — 2D point used by `document.Create.NewSpace(level, uv)`.
- `Line.CreateBound(XYZ p1, XYZ p2)` for finite lines; `Line.CreateUnbound` for rays.
- `Plane.CreateByNormalAndOrigin(XYZ normal, XYZ origin)` — then `SketchPlane.Create(doc, plane)` to host curves.

## Revit 2025 / 2026 version skew

`HVAKR.Revit.csproj` references Revit **2026** DLLs; `HVAKR.Revit.UI.csproj` references **2025**. The plugin is deployed to *both* 2025 and 2026 by the installer. That means:

- APIs removed in 2026 but present in 2025 → fine in `HVAKR.Revit.UI`, broken in `HVAKR.Revit`.
- APIs added in 2026 → off-limits in `HVAKR.Revit.UI`.
- Changes between point releases are rare but real (we've seen `GetProjectPosition`, `GetBoundarySegments`, `ActiveProjectLocation` shift around between 2023→2024→2025).

**Convention:** annotate version-gated calls with a trailing comment: `// 2025.3` or `// 2026`. When you add a new Revit API call, verify it against both 2025 and 2026 SDK docs.

Reference docs:
- Revit 2025 API: https://www.revitapidocs.com/2025/
- Revit 2026 API: https://www.revitapidocs.com/2026/

## Dialogs

User-facing dialogs inside a Revit context use `TaskDialog`:

```csharp
TaskDialog.Show("HVAKR", "Sync failed: " + ex.Message);
```

WPF UI code (the dockable pane UserControl) uses `MessageBox.Show`. Don't mix — `TaskDialog` doesn't work outside Revit (breaks `HVAKR.Revit.UI.Harness`), and `MessageBox` looks out-of-place in Revit.

## Don'ts

- Don't call Revit API from a WPF event handler. Always `.Raise()` an `ExternalEvent`.
- Don't mutate the document outside a `Transaction`.
- Don't change `DockablePaneGuid` or the addin GUID (`F6EF6882-...` in the installer).
- Don't call `doc.Regenerate()` or `uiDoc.RefreshActiveView()` "just to be safe" — Revit handles repaint after the transaction commits.
- Don't `Dispatcher.Invoke` Revit API calls — threading is about *Revit's* context, not WPF's.
- Don't capture `UIDocument` or `Document` references outside a single `Execute` call — they can become stale.
